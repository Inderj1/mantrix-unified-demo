Core Resolution Strategies
Six Ways to Resolve an Invoice Line to a PO Item
Each strategy contributes a weighted confidence score. Mantrix combines all available signals into a composite match probability for every possible invoice-line ‚Üí PO-item pairing.
Strategy 1 ‚Äî Weight: 30%
Key-Based Resolution
The invoice or OCR output contains a PO number and sometimes a PO line item number. If the vendor references the PO line directly ‚Äî immediate resolution. Also check for vendor material number in EINA-IDNLF (vendor's part #) and customer material number in MARA-BISMT. Many A&D vendors print SAP material numbers or PO line refs on their invoices.
EKKO-EBELN EKPO-EBELP EINA-IDNLF EINE MARA-BISMT
Resolution Logic
OCR extracts PO # ‚Üí validate against EKKO (vendor/company code match)
Extract PO line # if present ‚Üí direct match to EKPO-EBELP
Extract vendor part # ‚Üí look up EINA-IDNLF for this LIFNR ‚Üí get MATNR
If MATNR found ‚Üí match to EKPO-MATNR on this PO
Confidence: 98% if PO+line exact. 90% if material # resolved via EINA.
Strategy 2 ‚Äî Weight: 25%
Vendor-Material Translation Dictionary
This is the key differentiator. Over time, Mantrix builds a dictionary: "When vendor 1000456 (Thales) says 'Avionics Harness Assembly' they mean Material 500892-HA." Built from historical RSEG (invoice items) + EKBE (PO history) where invoices were successfully posted. This is exactly what a senior AP clerk has in their head.
RSEG-MATNR RSEG-TXZ01 EKBE RBKP-LIFNR
Resolution Logic
Query: For LIFNR 1000456, what MATNR was previously invoiced where RSEG-TXZ01 ‚âà "Avionics Harness"?
Build vendor-description‚Üímaterial map from all RSEG records for this vendor
Use fuzzy text matching (Levenshtein + TF-IDF) on invoice text vs RSEG-TXZ01 history
If same vendor + same description = same material 5+ times ‚Üí confidence 92%
Dictionary auto-updates with every successful posting. Self-improving.
Strategy 3 ‚Äî Weight: 15%
Semantic Description Matching
When no key and no history exists (new material, new vendor), AI uses semantic embeddings to compare invoice line text against PO item short text (EKPO-TXZ01), material description (MAKT-MAKTX), material long text, and PO item long text. Not string matching ‚Äî meaning matching. "Titanium Bolts 1/4-20" matches "HEX CAP SCREW Ti 0.25-20".
EKPO-TXZ01 MAKT-MAKTX STXL (long texts)
Resolution Logic
Generate embedding vector for invoice line text
Generate embedding vectors for all unmatched PO line texts (TXZ01 + MAKTX + long text)
Cosine similarity ranking of all PO lines against invoice line
Must exceed 0.72 threshold to be considered a candidate
Combine with qty/price proximity to boost or penalize (Strategy 4)
Strategy 4 ‚Äî Weight: 15%
Quantity & Price Proximity
Independent of text: what PO line has the closest quantity and unit price? Invoice says 12 EA at $3,780 ‚Äî PO Item 10 says 15 EA at $3,710. That's the closest fit by both qty and price, even though description doesn't match. Also accounts for partial deliveries by checking already-invoiced qty in EKBE.
EKPO-MENGE EKPO-NETPR EKBE-MENGE (already invoiced)
Resolution Logic
For each PO line: remaining qty = EKPO-MENGE ‚àí Œ£(EKBE-MENGE where BEWTP='Q')
Qty proximity = 1 ‚àí abs(invoice_qty ‚àí remaining_qty) / max(invoice_qty, remaining_qty)
Price proximity = 1 ‚àí abs(invoice_price ‚àí EKPO-NETPR) / max(invoice_price, EKPO-NETPR)
UOM check: EKPO-MEINS must be convertible to invoice UOM (T006 conversion table)
Composite = (qty_proximity √ó 0.5) + (price_proximity √ó 0.5). Must exceed 0.80.
Strategy 5 ‚Äî Weight: 10%
GR Cross-Reference
If a GR was posted, the GR document links material received to PO line item. Invoice qty matches GR qty ‚Üí strong confirmation signal. This is the "physical truth" check: we know what arrived because someone counted it and posted MIGO.
MATDOC / MSEG EKBE-BEWTP='E' EKET
Resolution Logic
For each PO line: get GR qty from EKBE where BEWTP='E' (GR) and BWART='101'
If GR qty = invoice qty for a PO line ‚Üí strong match signal (+15 confidence)
If no GR exists: check EKET-EINDT. Past delivery date + no MATDOC = GR pending
If multiple GRs: match to the GR document closest in date to invoice date
For service lines: check ESKN/ESLH (service entry sheets) instead of MATDOC
Strategy 6 ‚Äî Weight: 5%
Elimination & Constraint Satisfaction
After high-confidence matches are assigned, use process of elimination. If 3 of 4 invoice lines are matched, and 1 PO line is unmatched, and the amounts roughly fit ‚Üí assign. Also checks: total invoice amount ‚âà sum of matched PO line amounts (sanity check).
Derived from all above strategies
Resolution Logic
Run constraint satisfaction: invoice total ‚âà Œ£(matched PO line amounts)
Unmatched invoice lines ‚Üí unmatched PO lines with best amount fit
If remaining delta > tolerance ‚Üí flag as unplanned cost (freight, misc)
Accounts for split lines (1 PO line ‚Üí 2 invoice lines) and merged lines
Final confidence = weighted average of all strategies that contributed to this pairing

Guardrails & Hard Rules
Rules That Cannot Be Overridden
AI suggests. But some things are hard stops regardless of confidence score.
üõë
No GR = No Post (if WEBRE)
If PO item has GR-based IV (EKPO-WEBRE='X') and no GR exists in MATDOC/EKBE, the invoice CANNOT be posted. AI can park it and explain why.
üõë
Over-Invoice Protection
Invoice qty + previously invoiced qty (EKBE BEWTP='Q') cannot exceed PO qty + configured over-delivery tolerance (OMR6). Hard stop.
üõë
Duplicate Invoice Block
If RBKP shows same vendor + same REF_DOC_NO + same GROSS_AMOUNT within 365 days, block. No override without manager approval.
üõë
Blocked/Deleted PO Items
EKPO-LOEKZ ‚â† ' ' or EKPO-ELIKZ = 'X' ‚Üí cannot match to this line. AI must find an alternative or flag exception.
‚ö†Ô∏è
Price Tolerance (Configurable)
If price variance exceeds OMR6 tolerance group ‚Üí soft block. AI shows variance %, contract clause, precedent. Clerk can override with reason.
‚ö†Ô∏è
Low Match Confidence
If no line-item pairing exceeds 75% confidence ‚Üí don't pre-fill. Show candidate list to clerk. Never guess on line-item assignment below threshold.
‚ö†Ô∏è
UOM Mismatch
If invoice UOM cannot be converted to PO UOM via T006 ‚Üí flag for clerk. "Invoice says KG, PO says LB" ‚Äî AI converts if possible, else blocks.
üìù
Every Match Logged
For every line-item pairing, log: which strategies contributed, confidence scores, alternatives considered, and clerk's confirmation or override.
üìù
SAP Tolerance Groups
Read tolerance config from T169G / OMR6. Don't hardcode. Each company has different tolerances by company code, plant, and user group.
